Dbstl Typical Use Case

Among others, following are some typical use cases in which you would prefer to use dbstl rather than C++ STL:

  • You are handling a huge amount of data stored in C++ STL which can't reside in memory, a lot of OS page swap would be needed so that performance is degraded. When using dbstl, which is backed by Berkeley DB, the data is stored in a database and only needed portion is in memory, so that the overall performance of the machine is not slowed down.

  • You want to make use of Berkeley DB in a easier way, getting away with the many unfamiliar APIs and DBT marshalling/unmashalling details and DB structures/objects housekeeping work.

  • You want transaction semantics (ACID properties or a subset of ACID properties) in your memory structure access---you have a lot of options to make it closer in performance to memory structures.

  • You want full concurrency support to access C++ STL style containers. Few (if any) existing C++ STL implementations supports reading/writing to the same container concurrently, almost all of them only support accessing different containers of the same type concurrently.

  • You want to have object persistence in your application, and store objects in a database, and use the objects across different runs of your application. dbstl is capable of storing complicated objects which is not located on a continous memory chunk, with the help of some user configurations and callbacks.
