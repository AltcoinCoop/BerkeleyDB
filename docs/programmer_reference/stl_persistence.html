Direct Database Get

Each container's begin() method has a parameter "directdb_get", meaning whether directly read from database whenever the created iterator is used like this *itr or itr->member. If directdb_get is true(the default), then the iterator's operator* will always retrieve current fresh data from db, thus if you call 


*itr.datamember=new-value; 
*itr.datamember2=new-value2;

then the assignment to datamember is lost because the second assignment will use db's current key/data pair from database to assign to the iterator.

A correct alternative is like this: 


container::value_type &ref = *itr;
ref.datamember=new-value;
ref.datamember2=new-value2;
...// more member function calls and datamember assignments
ref._DB_STL_StoreElement();

The above code won't lose the new-value of the ref.datamember.

If you have two iterators to go through the container like this: 


for (itr1 = v.begin(), itr2 = v.begin(); itr1 != v.end(); ++itr1, ++itr2) {
	*itr1 = new_value;
	print(*itr2); // line 1
}

then the directdb_get parameter of begin() function must be set to true in the v.begin() calls otherwise line 1 prints the old value of *itr1.

Alternatively, you can set directdb_get to false and call itr2->refresh() immediately before line 1 , so that itr2's cached value is refreshed. The iterator::refresh() methods is designed this purpose, though this still can't completely replace direct database get, so the easiest way is to simply use the default true value of directdb_get.

If directdb_get is false, a few of the tests in dbstl's test kit will fail, because in the few algorithms of C++ STL the above special case happens, so by default directdb_get parameter in container::begin() methods is true. If your use case do not have such bizzar usage of iterators, you can set it to false to make iterator read operation faster.
