db_vector Specific Notes

  • You must share the same database handle among all threads of the same process to create container objects backed by the same database, and thus set DB_THREAD to all database open calls.

  • set DB_RENUMBER flag to the db handle if you want db_vector<> to work like std::vector or std::deque; Don't set DB_RENUMBER if you want db_vector<> to work like std::list so that it is faster.

    For example, to construct a queue/stack object, you only need a db_vector<> object whose database handle is not set the DB_RENUMBER. Of course if the db handle has DB_RENUMBER it still works for this kind of scenario, but it is not as fast.

    db_vector will not check whether DB_RENUMBER is set or not, if you didn't set it and expect db_vector<> to work like std::vector<>/std::deque<>, you will get wrong results in its operator[], because the indices are not maintained any more.

    There is example code to use this feature in TestVector::test_queue_stack method of dbstl test suite.

  • Just as std::vector, inserting/deleting in the middle of the sequence is slower than doing the same action at the end of the sequence because the underlying DB_RECNO type of db with DB_RENUMBER flag set works slow when inserting/deleting in middle or head --- it has to update all those records's index after the inserted/deleted one. If you don't need to keep the index renumbered on insert/delete, you can use db_map instead.

    db_vector also contains methods of std::list and std::deque, including std::list<>'s unique methods including remove, remove_if, unique, merge, sort, reverse, and splice, with identical semantics/behaviors, though pushing/deleting front is slow for std::deque equivalent when there are quite a lot of elements in the database.

  • 6. You can use std::queue, std::priority_queue and std::stack container adapters, they work with db_vector even without DB_RENUMBER set.

Associative Container Specific Notes

db_map contains the union of method set of std::map and hash_map, but there are some methods that can only be called on containers backed by DB_BTREE or DB_HASH databases, so you can call db_map<>::is_hash to figure out what type of db it is backed, and avoid calling unsupported methods, doing so will cause an InvalidFunctionCall exception.

DB_BTREE specific methods: upper_bound(), lower_bound(), key_comp(), value_comp(); DB_HASH specific methods: key_eq(), hash_funct().
